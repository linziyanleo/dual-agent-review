# Plan v1: 给 /api/login 加 token-bucket rate limit

## Context / Goals
- 当前 `/api/login` 无任何 rate limit，被 brute-force 撞过几次（incident #INC-412）。
- 目标：把对单个 IP 的登录尝试限制到 **每分钟 10 次**；超限返回 `429 Too Many Requests`。
- 不影响正常用户体验（人类 1 分钟内不会试 10 次密码）。

## Non-goals
- 不做 CAPTCHA、不接 WAF —— 那是另一张工单。
- 不做按用户名 rate limit（先按 IP，下个迭代再加 username 维度）。
- 不做分布式 rate limit（当前后端只跑单实例）。

## Proposed approach
1. 在 `internal/middleware/ratelimit.go` 实现 `TokenBucket` struct：
   - `capacity = 10`、`refill = 10 tokens/min`
   - 用 `sync.Mutex` 保护 `lastRefillAt + tokens`
2. 在 `internal/middleware/ratelimit.go` 实现 `IPRateLimiter` —— 内部 `map[string]*TokenBucket` + RWMutex。
3. 把 limiter 挂到 `/api/login` route：`router.POST("/api/login", ratelimit.ByIP(perMinute=10), handler.Login)`
4. 超限分支：return `c.JSON(429, gin.H{"error": "too many login attempts"})`。

## Affected files
- `internal/middleware/ratelimit.go`（新建）
- `internal/middleware/ratelimit_test.go`（新建）
- `cmd/server/routes.go`（挂中间件）

## Risks & open questions
- map 会无限增长（没 IP 过期机制）—— v2 可以加，但 v1 先用内存够。
- 进程重启 token 计数清零 —— 攻击者可以触发崩溃 + 重启绕过。先记 risk，不处理。
- IP 取值：直接用 `c.ClientIP()`，假设 reverse proxy 设了 `X-Forwarded-For`。

## Verification plan
- 写 `ratelimit_test.go`：连发 11 次，第 11 次返回 429。
- 写 `ratelimit_test.go`：等 6.1 秒后能续 1 次（10 tokens/min = 1 token / 6s）。
- 手动 `curl` 11 次确认前端报错合理。
